# リポジトリの目的

コーディンテスト対策を行う

# 言語・環境

- Rustを使用する
- 環境構築方法や新規問題の作成方法は README.md を参照

# 注意するべきこと

- 実装ファイルは指示がない限り編集しない
- 実装のヒントを求められた時は具体的な実装を提示せず、アルゴリズムやデータ構造の方針を最小の情報で提示する
- ヒントではなく、解答を求められた場合は具体的な実装を提示、またはファイルの編集を求められれば編集する

# レビュー規約

レビューを依頼された場合は、次の2つの観点を両方含める。

- CS 観点: アルゴリズム、計算量、データ構造、境界条件、問題設定との整合性を見る。
- Rust 観点: 所有権、借用、型、命名、標準ライブラリの使い方、Rust らしい表現を見る。

指摘は重要度の高いものから並べ、必要に応じてファイルと行番号を示す。
Rust 観点では、コンパイルできるかだけでなく、より Rust らしく安全・簡潔に書ける標準的な形がある場合も指摘する。例: `while !queue.is_empty()` と `pop_front().unwrap()` の組み合わせより、`while let Some(id) = queue.pop_front()` の方が自然な場合はその理由とともに提案する。
CS 観点では、実装が正答するかだけでなく、採用している状態・データ構造・不変条件がアルゴリズム上どんな意味を持つかまで踏み込む。たとえば BFS/DFS の `color` が単なる訪問済み判定としてしか使われていない場合は、この問題では `distance` や `visited` で代用できること、逆に DFS の閉路検出・トポロジカルソート・探索中/処理済みの区別が必要な場面では `White/Gray/Black` が有効に働くことを説明する。
学習用コードでは、`new()` に対応する `Default` 実装がないことだけを独立したレビュー指摘にしない。
ただし、その関数境界とは別に、全トークン・全操作・全結果などの大きな中間データを重ねて保持している場合は、問題の制約や学習目的に照らして必要に応じて指摘する。
公式サンプル以外のテストケースを用意できない問題では、追加テストの不足だけを主要な指摘にせず、境界条件や不変条件をコードから丁寧に読み取ってレビューする。
